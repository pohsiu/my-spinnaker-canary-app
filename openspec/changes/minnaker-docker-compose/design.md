## Context

`spinnaker/dinghyfile`, `spinnaker/canary-config.json`, and the new `spinnaker/frontend-dinghyfile` / `spinnaker/micro-frontend-dinghyfile` are pipeline-as-code files that have only ever been JSON-syntax-checked in this repo — none have run against an actual Spinnaker + Kayenta install. [Minnaker](https://github.com/armory/minnaker) is Armory's official lowest-friction Spinnaker installer, but it's a root shell script (`install.sh`) that assumes a bare Linux VM: it installs k3s directly on the host, then (per its actual `readme.md` at the pinned tag, not assumed) clones a *separate* repo, [`armory/spinnaker-kustomize-patches`](https://github.com/armory/spinnaker-kustomize-patches) (branch `minnaker`), and applies it with `kubectl apply -k` via the Spinnaker Operator. The `minnaker` repo itself vendors no Spinnaker/datastore manifests of its own beyond stale Halyard-era leftovers — the real, current manifests live in that second repo. This repo runs everything else (backend, and the CDN tooling image) through Docker, so a bare-VM installer doesn't fit the existing "everything is a container" workflow — and it isn't portable to a contributor's laptop the way `docker compose up` is.

Two things confirmed by reading the actual pinned sources (not assumed from minnaker's marketing copy):
- **Kayenta/Prometheus is NOT enabled by default** in `spinnaker-kustomize-patches`'s `minnaker` recipe (`recipes/kustomization-minnaker.yml`) — the canary and Prometheus-Grafana resources are present in the repo but left commented out with a `# TODO - investigate what changes... need to support canarying`. Getting Kayenta working is a deliberate addition on top of the minnaker recipe, using that same repo's own (otherwise-unused) `accounts/canary/prometheus.yml` and `infrastructure/prometheus-grafana/` — not something minnaker hands you "for free."
- Upstream maintenance is uneven across the three repos this design pulls from — worth stating once rather than re-litigating per decision: `armory/minnaker` is fairly dormant (last tag `v0.1.3`, 2021-04-30; last commit, 2023-05-20; not archived) and is used here only as a reference for *what to reproduce*, not as a dependency we fetch from. **Correction, found during implementation**: `armory/spinnaker-kustomize-patches`'s *repo* shows recent push activity, but that's on its default branch — the `minnaker` branch we actually pin to has not been touched since **2021-03-26** (checked directly via the GitHub API's per-branch commit list, not the repo-level `pushed_at` I'd relied on here originally). `armory-io/spinnaker-operator` (the Armory-flavor operator we actually use, confirmed public) is the one repo here that's genuinely current (`v1.8.11`, published 2025-12-04).

Local dev machine currently observed with ~11GB free disk (see `CLAUDE.md`), against minnaker's stated 30GB minimum — this design doesn't attempt to shrink the stack to fit; a real Spinnaker install has a real footprint, and a smaller-than-real footprint would make it a misleading test harness. The constraint is called out for whoever runs this, not engineered around.

## Goals / Non-Goals

**Goals:**
- Reproduce minnaker's actual stack (k3s, Minio, MySQL, Redis, Spinnaker Operator, `SpinnakerService`) as a `docker-compose.yml` under `others/minnaker/`, runnable with `docker compose up` on any Docker host with privileged-container support — plus Kayenta and a Prometheus canary account layered on top using `spinnaker-kustomize-patches`'s own (upstream-provided but disabled-by-default) canary resources, since minnaker's default recipe doesn't include a working canary setup.
- Make it disposable: `docker compose down -v` leaves nothing behind.
- Document, in `others/minnaker/README.md`, exactly how far this gets a contributor toward testing `spinnaker/*dinghyfile` — including what's still a placeholder afterward.

**Non-Goals:**
- Wiring Prometheus to `apps/backend`'s `/metrics` or deploying this repo's Helm chart into the resulting cluster — that's a real next step, but a separate change, since it requires deciding how `apps/backend`'s Docker image gets into a registry this local k3s can pull from.
- Making the stack lightweight enough to run in the ~11GB currently free on this dev machine. Out of scope by design (see Context).
- A hosted/CI-integrated version of this — it's a local, manual testing aid, not part of `.github/workflows/`.
- Reimplementing minnaker's actual `install.sh` logic from scratch; this wraps the same upstream artifacts (k3s image, Spinnaker Operator manifests) minnaker itself uses, at a pinned version.

## Decisions

**k3s-in-Docker via the official `rancher/k3s` image, run `--privileged`, rather than a bare-VM install or a wrapper CLI (k3d/kind).**
Minnaker's own value proposition is "k3s is lighter than full k8s" — matching that keeps this stack's resource footprint close to what minnaker itself needs, unlike `kind` (upstream k8s, heavier). `k3d` is a thin wrapper that does approximately this same `docker run` under the hood; running the `rancher/k3s` image directly in `docker-compose.yml` keeps the whole stack in one inspectable, version-controlled file instead of shelling out to another CLI tool as a dependency.

**A one-shot "bootstrap" container (not baked into the k3s image) installs the Spinnaker Operator CRDs/Deployment, then runs a single `kubectl apply -k others/minnaker/manifests/` that pulls in minnaker's upstream recipe as a remote kustomize base, polling `kubectl get nodes` until the k3s server is ready first.**
Keeps the k3s server container itself unmodified from upstream (easier to bump versions later). `kustomize` (built into `kubectl` since 1.14) supports remote bases pinned by commit — `resources: [github.com/armory/spinnaker-kustomize-patches/recipes?ref=<sha>]` — so the bootstrap container doesn't need its own `git clone` step or a vendored copy of minnaker's Minio/MySQL/Redis/`SpinnakerService` manifests; `kubectl` fetches them directly from the pinned upstream ref at apply time.

**Manifests are minnaker's own defaults, pinned to a specific commit of `spinnaker-kustomize-patches`'s `minnaker` branch — not a `minnaker` repo release tag, and not hand-written from scratch.**
The `minnaker` repo's own release tags (latest: `v0.1.3`) don't correspond to fixed manifest content — the manifests minnaker's installer actually applies live in `spinnaker-kustomize-patches`, a separately-versioned repo with no tags of its own. Pinning a commit SHA on its `minnaker` branch (`50a095cd9aa1a83723f4cc09d6e5a78c6d96d17e` as of this design) is the actual reproducibility anchor — bumping it later is a deliberate, visible change to `others/minnaker/manifests/kustomization.yml`.

**Kayenta + Prometheus are enabled via one small local patch file (`others/minnaker/manifests/patch-canary-prometheus.yml`) added as an extra `resources`/`patchesStrategicMerge` entry on top of minnaker's base recipe — not authored from scratch.**
`spinnaker-kustomize-patches` already ships `accounts/canary/prometheus.yml` (sets `canary.enabled: true` + registers a `prometheus` metrics account) and `infrastructure/prometheus-grafana/` (an in-cluster Prometheus+Grafana deployment) — minnaker's own `kustomization-minnaker.yml` just doesn't reference them (left commented out, marked TODO). Our local `kustomization.yml` adds both as extra remote-base resources, plus a copy of `accounts/canary/prometheus.yml` with `baseUrl` changed from the placeholder `http://myprometheus` to the real in-cluster service (`http://prometheus.spinnaker:9090`, confirmed from `prometheus-grafana.yml`'s `Service` definition) — the one deliberate content change from upstream, not a placeholder we're leaving for someone else to fill in, since the target is a same-cluster Service we control.

**Cluster access from the host via a `kubectl port-forward` sidecar container, not k3s's built-in Traefik ingress.**
Minnaker (on a real VM) exposes Spinnaker through Traefik on the VM's public IP. There's no public IP here — the simplest way to reach Gate (API, :8084) and Deck (UI, :9000) from the host is a small container in the compose network that holds a `kubectl port-forward` open and publishes those ports through Docker's own port mapping. Avoids configuring/trusting a second ingress layer for a disposable local stack.

## Risks / Trade-offs

- **[Risk] `--privileged` k3s-in-Docker won't run on restricted Docker hosts** (rootless Docker, some CI runners, some corporate-locked-down Docker Desktop policies) → Mitigation: documented as a hard prerequisite in `others/minnaker/README.md`, no fallback attempted — the alternative (real VM) is what minnaker already covers.
- **[Risk] Real resource footprint (4 vCPU / 16GB RAM / 30GB disk) won't run on this repo's current dev machine** (~11GB free disk observed) → Mitigation: documented explicitly rather than silently failing partway through `docker compose up`; README states the requirement before the quickstart steps, not after.
- **[Risk] Pinned manifests drift from upstream `spinnaker-kustomize-patches` over time** (security patches, new Operator versions, upstream finally enabling canary by default and making our patch redundant or conflicting) → Mitigation: the pinned commit SHA is named in the README and in `manifests/kustomization.yml`'s `ref=` as the thing to bump; not auto-tracked. If upstream ever un-comments the canary resources in its own recipe, our local patch becomes redundant — worth a periodic check, not solved here.
- **[Trade-off] Doesn't validate against `apps/backend`'s real metrics or Docker image** — only the Spinnaker/Kayenta control plane comes up; a `dinghyfile` run through it still needs a reachable registry and a Prometheus scrape target pointed at real canary/baseline pods. Accepted as a deliberately separate follow-up (see Non-Goals) rather than scope-creeping this change into "also deploy the whole app."

## Migration Plan

No existing system is modified — this is a net-new, self-contained addition under `others/minnaker/`.

1. `docker compose -f others/minnaker/docker-compose.yml up -d`
2. Bootstrap container polls until k3s is ready, installs the Spinnaker Operator, then runs `kubectl apply -k others/minnaker/manifests/` (fetches minnaker's base recipe + the canary/Prometheus additions from the pinned upstream commit, applies the local `baseUrl` patch), then polls the `SpinnakerService` resource until healthy and exits 0.
3. Once Gate/Deck are reachable on the forwarded host ports, a contributor manually applies a pipeline JSON (e.g. `spinnaker/dinghyfile`'s contents) directly via the Spinnaker API to test pipeline execution — see the resolved Open Question below for why this is manual rather than wiring up Dinghy.
4. Teardown: `docker compose -f others/minnaker/docker-compose.yml down -v` — no state survives outside the compose project's named volumes, which that command removes.

## Open Questions

- ~~Should the bootstrap container also install Dinghy (so `dinghyfile` changes sync automatically)?~~ **Resolved: no.** Wiring Dinghy means standing up its own webhook listener and pointing it at a git remote it can poll — real setup for a disposable test harness whose whole point is being thrown away. Manually applying pipeline JSON via the Spinnaker API is one `curl` call and keeps the manifest set smaller; revisit only if that proves too tedious in practice across repeated test runs.
- ~~Exact minnaker release tag to pin~~ **Resolved: not a `minnaker` tag at all.** The manifests that matter live in `spinnaker-kustomize-patches`; pinned to commit `50a095cd9aa1a83723f4cc09d6e5a78c6d96d17e` on its `minnaker` branch (current HEAD as of this design — that branch's last commit was 2021-03-26, not recent; see the corrected note in Context). See Decisions.
- ~~`SpinnakerService` creation rejected: "no account of type CONFIGURATION_STORE and OBJECT_STORE configured"~~ **Resolved.** Not an Operator-version mismatch as first suspected (that theory was wrong — `persistence/patch-minio.yml` covers `persistentStorage`, a separate concern from what Kayenta itself needs). The actual cause: Kayenta requires its *own* canary-scoped account with these `supportedTypes`, separate from `persistentStorage`, to store canary configs/results — and upstream's `accounts/canary/` folder has no S3/Minio example at all (only Datadog, Dynatrace, Stackdriver, Prometheus), unlike every other gap in this design which upstream at least demonstrated somewhere. Fixed by adding an `aws`/Minio canary service integration modeled on Armory's own docs. Along the way, also fixed a real, unrelated blocker (`armory/terraformer` has no ARM64 image and was failing the aggregate health check) by disabling Terraform integration. End state, confirmed live: `SpinnakerService is healthy`, Gate and Deck both respond, a real browser session through Deck's login successfully made authenticated Gate API calls. See `others/minnaker/README.md`'s "Current status" for full detail.
- The old-Operator dead end (`v1.2.5`, webhook crashes against a modern k8s API — compiled-in, unpatchable) remains correctly ruled out; `v1.8.11` is what actually works, and was never the problem.
