01

目的

這個 repo 不是一個要上線的產品，而是一份「可以跑起來的教材」—— 示範漸進式交付（progressive delivery）的完整鏈路：程式碼變更如何 被打包、部署成獨立的 canary 版本、由真實指標自動判定是否安全， 再決定要不要晉升成正式版本，全程不需要人工盯著儀表板做決定。

三個部分刻意分開又刻意串連：Web 服務提供會被觀測的真實流量與指標； Helm Chart 讓同一份程式碼可以用 track 參數部署成 production / baseline / canary 三個獨立且不互相干擾的版本； Spinnaker Pipeline-as-Code 把「部署 → 分析 → 晉升」這個決策流程 寫成版本控制得到的設定檔，而不是點點點的手動操作。

02

架構總覽

六個各自獨立、透過慣例（convention）串接的部分：

apps/frontend Vite + React SPA · host

開發時走 HMR dev server；上線產出純靜態 dist/，執行期透過 Module Federation 載入 micro-frontend。

apps/micro-frontend Vite + React SPA · remote

曝出 remoteEntry.js，被 apps/frontend 在 執行期載入，不是 build-time 依賴。

apps/backend Express + prom-client

提供 /api/*、暴露 /metrics 給 Prometheus，正式環境下同時 serve 前端靜態檔。

helm/ 單一 Chart，三種 track

用同一份 template，靠 --set track= 產生互不衝突的 production / canary / baseline 資源（backend 專用）。

spinnaker/ 3 條 Pipeline-as-Code

backend 用 Helm/Kayenta；frontend、micro-frontend 各自獨立的 CDN 部署 pipeline。

cdn/ Cloudflare Pages + Workers 設計

frontend、micro-frontend 的部署目標；Worker 做金絲雀流量分流。

串接方式 Prometheus 依 Pod annotation（prometheus.io/scrape）自動抓取 /metrics； Kayenta 再用 track label 把同一份 PromQL 查詢分別套用到 canary 與 baseline 兩組 Pod 上做比較—— 整條鏈路沒有任何一段是寫死在程式碼裡的，全部靠 label/annotation 這種 Kubernetes 慣例動態接起來。
03

本地開發

Turborepo 同時啟動兩個 app：Vite dev server（:5173，HMR） 把 /api 與 /metrics proxy 到 Express（:3000）。 開發時感受不到後端存在，但呼叫的其實是同一支 server.js。 

npm install
npm run dev
# → Vite   http://localhost:5173  (HMR)
# → Express http://localhost:3000 (/api, /metrics)
04

打包與 Docker

正式環境不跑 Vite，只跑 Express。前端先被打包成純靜態檔， Express 直接把它當 SPA serve 出去，非 /api、/metrics 的路由 一律回傳 index.html，交給前端 router 處理。 

npm run build --workspace=frontend
npm start
# Express 監聽 :3000，serve apps/frontend/dist

apps/backend/Dockerfile 用 multi-stage build：第一階段裝整個 workspace 只為了跑 vite build，第二階段只把 Express 依賴 + 打包產物 放進最終 image，體積不受前端 devDependencies 拖累。 

# 從 repo 根目錄建置
docker build -f apps/backend/Dockerfile -t my-spinnaker-canary-app .
05

Helm Chart：三個互不衝突的 track

整個 canary 機制能成立，關鍵在於 Deployment / Service 的名稱與 selector 都吃 .Values.track，所以同一份 Chart 部署三次會產生三組完全獨立的資源：

production

正式流量。晉升階段會把新版本以這個 track 重新部署，取代舊版本。

canary

帶著新版 image 的少量副本，只在分析期間存在，接受和 baseline 對照的觀測。

baseline

用「舊版但同時新建」的副本，排除掉「重啟造成的雜訊」，讓 canary 的比較基準更乾淨。

helm template my-app helm/ --set track=canary --set image.tag=pr-123
# → Deployment/Service: my-app-canary
#   label track=canary，不會和 my-app-production 互相覆蓋
06

Spinnaker Pipeline-as-Code（Dinghy）

spinnaker/dinghyfile 是宣告式的流水線定義，被 Dinghy 監看並自動同步成 Spinnaker 裡的 pipeline——流水線本身也是版本控制的一部分。三個階段：

TRIGGER

Docker Registry

新 image tag 被 push 時觸發流水線。

STAGE 1

Bake Manifest

用 Helm3 渲染 Chart，以 track: canary 部署新版本。

STAGE 2 · GATE

Kayenta 分析

30 分鐘觀測窗，比對 canary 與 baseline 指標，算出分數。

STAGE 3

Promote

分數通過門檻後，以 track: production 重新部署，完成晉升。

第二階段是唯一的決策點：分數沒過，流水線就停在這裡，正式環境完全不受影響—— 這正是「金絲雀部署」相對藍綠部署的核心優勢：壞版本只影響一小群 canary Pod， 而且是被指標主動擋下來的，不需要人工介入。

07

Kayenta：兩個指標，一個分數

spinnaker/canary-config.json 定義了兩條 PromQL， 分別對 track=canary 和 track=baseline 的 Pod 各查一次再比較：

指標PromQL判定方向權重
HTTP 5xx Error Rate rate(http_requests_total{code=~"5.."}[2m]) canary 上升即扣分（critical） 100
HTTP p95 Latency histogram_quantile(0.95, ...) canary 上升即扣分 100

綜合分數對應三種結果，門檻寫在 dinghyfile 裡：

分數結果流水線行為
≥ 90 PASS 自動晉升到 production
75 – 89 MARGINAL 標記待確認，不自動晉升
< 75 FAIL 流水線終止，canary 回收
08

Micro-frontend（Module Federation）

apps/frontend 不再是唯一的前端。apps/micro-frontend 是獨立打包、獨立部署的 第二個 SPA，透過 vite-plugin-federation （Vite 版 Module Federation）在執行期被 host 載入——兩者的 build 完全不互相依賴。

兩欄從原始碼到部署完全獨立、互不阻塞；唯一的連接點是最下面那條虛線——瀏覽器執行期向 micro-frontend-router 要 remoteEntry.js，不是 frontend 幫忙轉發，也不是 build 時就決定死的依賴。

remote · apps/micro-frontend exposes ./Widget

vite build 產出 remoteEntry.js——host 在瀏覽器裡動態 import() 的進入點。

host · apps/frontend remotes: micro_frontend

React.lazy(() => import('micro_frontend/Widget'))，外包一層 error boundary，因為 remote 現在是網路依賴而非本地 import。

npm run build --workspace=micro-frontend
npm run preview --workspace=micro-frontend   # 在 :4174 serve remoteEntry.js

MF_REMOTE_URL=http://localhost:4174/assets/remoteEntry.js \
  npm run build --workspace=frontend         # 把 remote URL 編進 host bundle
Remote 若掛掉 RemoteBoundary（error boundary + Suspense）會攔下載入失敗，顯示 「micro-frontend unavailable」而不是讓整個 host 白屏——micro-frontend 掛掉只影響它自己那塊 UI。
09

CDN 部署策略（frontend + micro-frontend）

兩個 SPA 都是純靜態產出，各自獨立部署到 Cloudflare Pages + Workers， 跟 backend 的 Helm/k8s/Kayenta pipeline 是完全平行、互不相依的另一條路。 （這裡原本設計是 AWS S3 + CloudFront，後來改選 Cloudflare——免費方案不需要信用卡， Pages 每次部署自帶版本化 URL，比手動維護 S3 版本前綴簡單。）

為什麼不能沿用 Kayenta Kayenta 現有的分析能成立，是因為 k8s 給了兩組可比較的 Pod（track=canary vs track=baseline），Prometheus 才能用 track label 分別查詢。CDN 上的 SPA 沒有 Pod，「健不健康」是 RUM／Web Vitals／JS error rate 的問題，不是 PromQL 能回答的問題—— 而這個 repo 目前還沒有 RUM 蒐集管線，所以下面用 Manual Judgment 頂著， 標記為未來要換成自動化分析的位置。

路由：Worker + Workers KV

cdn/edge/canary-router.js 是一個 Cloudflare Worker，每個 request 依 cookie 做 sticky 分流決定 stable / canary，再 fetch() 到對應的 Pages 部署 URL 當 proxy——一個 Worker 就處理完請求跟回應，不像先前 CloudFront Functions 設計需要拆成 viewer-request / viewer-response 兩支。每個 app 各自部署 一份同源的 Worker（wrangler.frontend.toml / wrangler.micro-frontend.toml 設定各自的 APP_NAME）。

KV Key（每個 app 各自命名空間）意義
<app>.stableUrl多數訪客看到的 Pages 部署 URL
<app>.canaryUrl金絲雀切分到的 Pages 部署 URL
<app>.canaryWeight0–100，新訪客被分到 canary 的機率

部署流程

spinnaker/frontend-dinghyfile 與 spinnaker/micro-frontend-dinghyfile，兩個 app 相同的五個階段：

STAGE 1

建立部署

wrangler pages deploy 產生新的部署 URL，不影響任何現有流量。

STAGE 2

切 10% 流量

更新 KV 的 canaryUrl / canaryWeight。

STAGE 3 · GATE

Manual Judgment

人工檢查 canary 切片；未來換成 RUM 指標自動判定。

STAGE 4

Promote 100%

stableUrl 切到新部署，canaryWeight 歸零。

STAGE 5

清邊緣快取

Pages 部署本身免清快取；purge 自訂網域上快取的 index.html。

兩條 pipeline 各自獨立部署，但 micro-frontend 若改了曝出的元件 API， 還是會讓當時線上的 host 版本壞掉——這點目前只能靠人工協調（Manual Judgment 階段互相確認）， 沒有 contract test 把關。細節見 cdn/README.md。

10

端到端流程

把上面每一段接起來，一次程式碼變更的完整路徑：

1. 開發者 push commit 2. CI 建置 apps/frontend + apps/backend → 用 apps/backend/Dockerfile 打成單一 image → push 到 registry 3. Registry 上出現新 tag → 觸發 spinnaker/dinghyfile 定義的 pipeline 4. Spinnaker 用 helm/ bake 出 track=canary 的 manifest 並部署 5. Prometheus 依 annotation 自動開始抓 canary Pod 的 /metrics 6. Kayenta 以 spinnaker/canary-config.json 比對 canary / baseline 30 分鐘 7. 分數 ≥ 90 → 以 track=production 重新部署，晉升完成 分數 < 75 → 流水線終止，正式環境未受影響
11

部署前必須設定的值

這份 repo 目前是「可運作但未指向真實環境」的狀態，以下都是 placeholder，接上真正的環境前需要換掉：

  • helm/values.yaml → image.repository：換成實際的 container registry 路徑
  • spinnaker/dinghyfile → triggers[].repository：需與上面同一個 registry 路徑
  • spinnaker/dinghyfile / canary-config.json → metricsAccountName：換成 Spinnaker 裡設定好的 Prometheus 帳號名稱
  • spinnaker/frontend-dinghyfile / micro-frontend-dinghyfile → <CDN_DEPLOY_TOOLS_IMAGE>：build 並 push cdn/deploy-tools/Dockerfile 後填入
  • cdn/edge/wrangler.*.toml → <CF_ZONE_ID>、<FRONTEND_DOMAIN> / <MICRO_FRONTEND_DOMAIN>（目前用共用的 workers.dev 子網域跑，換成自訂網域才需要這幾個）
CDN 已實際部署並驗證 兩個 Pages 專案、一個 Workers KV namespace、兩個 canary-router Worker 都已建立在真實 Cloudflare 帳號上（免費方案，免信用卡）。用 Playwright 對真實網址驗證過：host 透過 router 載入 remoteEntry.js 並渲染 widget；把 micro-frontend.canaryWeight 在 KV 裡調到 100% 後，新訪客真的看到 canary 版本，調回 0% 後恢復看到 stable 版本——不是只驗證 KV 寫入成功，是連著實際頁面載入結果一起驗證。過程中還抓到一個真的設計錯誤：MF_REMOTE_URL 原本指向固定的 Pages 部署網址，導致調整 canary 權重對 host 完全沒有作用，已修正為指向 router。
尚未驗證 本機沒有 Docker daemon、Kubernetes 叢集。backend 這邊 helm lint / helm template 已通過語法與樣板檢查，但沒有真實叢集可用；cdn/scripts/*.sh 沒有逐字跑過（測試時用 wrangler/KV 指令直接下，因為本機是 OAuth 登入，腳本刻意要求 CLOUDFLARE_API_TOKEN 給 CI 用）；還沒有自訂網域，目前跑在共用的 workers.dev 子網域上。

13

Retro：實際部署 Cloudflare pipeline 學到的事

以下是把 cdn/ 真的部署到 Cloudflare 帳號後留下的筆記—— 這些是設計階段容易想不到、但實際跑過就會馬上撞到的事。

設計「看起來會動」不代表真的會動 KV 裡 canaryWeight 的寫入全部成功，單獨看資料完全正確。 問題出在更上層：MF_REMOTE_URL 原本指向固定的 Pages 部署網址， 而不是 router，導致 host bundle 根本不會重新解析 remote——調整 canary 權重 對 host 完全沒有作用，而且不會有任何錯誤訊息。只有實際切換權重、重新整理頁面 才抓得到，光看 KV 的值或設定檔看不出來。
供應商選擇要在設計前定案 AWS S3+CloudFront 在還沒部署任何一行 infra 之前就被換成 Cloudflare—— 免費方案要不要信用卡，這種帳務層的限制應該在設計具體的 provider-specific 機制（KVS schema、Function 語法）之前就先決定，不是設計完才發現卡關。
免費方案的一次性互動步驟，agent 沒辦法代勞 wrangler login（瀏覽器 OAuth）跟註冊 workers.dev 子網域，都需要人類手動互動一次。規劃流程時要留這個 checkpoint， 不能假設 CLI 自動化可以包辦全部設定。
剛註冊的網域，TLS 憑證需要時間生效 新註冊的 workers.dev 子網域，部署當下馬上成功， 但實際發請求會先撞到 TLS handshake 失敗，大約一分鐘後憑證才生效可用。 「部署成功」跟「流量真的能打進來」是兩件事，尤其是全新的網域。
Kayenta 的模型套不到 CDN SPA，不是換個小地方就好 不是「一樣的概念、換個基礎設施而已」。沒有 Pod 就沒有 Prometheus scope， 沒有 scope 就沒有自動化分數，沒有模糊地帶。這裡的 pipeline 用 Manual Judgment 頂著，而不是硬凹一個看起來像但實際上答不了「這版本健不健康」的指標。
